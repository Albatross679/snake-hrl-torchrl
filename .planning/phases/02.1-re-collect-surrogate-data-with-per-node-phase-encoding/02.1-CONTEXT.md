# Phase 2.1: Re-collect Surrogate Data with Per-Node Phase Encoding - Context

**Gathered:** 2026-03-10
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace the Phase 1 dataset (2.3 GB, 28 batches) with a new dataset in a fundamentally different format: 2000-substep collection runs with intermediate state checkpoints, enabling overlapping training pairs with per-element CPG phase encoding. Covers: new collection pipeline, new batch file format (checkpoint arrays), and the dataset class that forms (state, phase_input, next_state) pairs on-the-fly. Does NOT cover: surrogate model architecture, training loop changes beyond the dataset class, or Phase 3 training.

</domain>

<decisions>
## Implementation Decisions

### Core format change: overlapping substep pairs
- Each "collection step" runs 2000 Elastica substeps with ONE fixed RL action (vs 500 currently)
- States saved every 10 substeps → 201 checkpoints per collection step (substep indices 0, 10, 20, ..., 2000)
- Valid training pairs: (checkpoint i, checkpoint i+50) = window of 500 substeps = 0.5s, same action throughout
- This yields 151 valid pairs per collection step (i = 0, 1, ..., 150 checkpoint indices)
- The action does NOT change mid-window — all pairs from one run have the same action

### Per-element phase encoding (replaces global [sin(ωt), cos(ωt)])
- Drop the 2 global phase features; replace with 60 per-element features for 20 elements:
  - [sin(k·s_j + ωt + φ), cos(k·s_j + ωt + φ)] for j=1..20 → 40 features
  - Commanded curvature kappa(s_j) = A·sin(k·s_j + ωt + φ) for j=1..20 → 20 features
- Where: k=wave_number (action[2]), s_j=element arc-position (fixed 0→1), ω=2πf (from action[1]), φ=phase_offset (action[3]), A=amplitude (action[0])
- New surrogate input dim: 124 (state) + 5 (action) + 60 (per-element) = 189 (vs current 131)
- Computed on-the-fly in the dataset class from saved (action, t_at_checkpoint) — not stored in batch files

### Raw checkpoint batch format (pair at training time)
- Each batch file saves raw arrays, NOT precomputed pairs:
  - `substep_states`: (N_episodes × N_steps, N_checkpoints=201, STATE_DIM=124) float32
  - `actions`: (N_episodes × N_steps, ACTION_DIM=5) float32
  - `t_start`: (N_episodes × N_steps,) float32 — accumulated sim time at start of each 2000-substep run
  - `episode_ids`, `step_ids`: metadata arrays
- Training dataset class forms pairs on-the-fly: (state[i], encode_per_element(action, t_start + i·10·dt_sub), state[i+50])
- Flexible: changing K or phase encoding formula does not require recollection

### Dataset class (in Phase 2.1 scope)
- New `OverlappingPairDataset` in `aprx_model_elastica/dataset.py`
- Loads checkpoint batch files, iterates over (run, checkpoint_i) index pairs
- Computes 189-dim input = state + action + per-element phase on-the-fly
- Compatible with existing WeightedRandomSampler and density weighting
- Old `SurrogateDataset` (Phase 1 format) kept but clearly marked deprecated

### Omega perturbation fix
- `perturb_omega_std`: 0.05 rad/s → 1.5 rad/s
- Motivation: operational omega_z is ~1–10 rad/s; old value gave negligible coverage of realistic angular velocity states
- Other perturbation params unchanged (position_std=0.002, velocity_std=0.01, curvature_max=3.0)

### Dataset handling
- Archive Phase 1 data: move (or rename dir to) `data/surrogate_v1/`
- Collect Phase 2.1 data fresh to `data/surrogate/` (same location, clean slate)
- Target: 25 GB of checkpoint data
- Keep Sobol quasi-random actions and 30% perturbation fraction from Phase 1 — those are still good
- Do NOT collect forces (no `collect_forces` field) — reduces file size; forces not used in surrogate training

### Claude's Discretion
- Exact `perturb_omega_std` value in [1.0, 2.0] range — 1.5 is a reasonable midpoint
- Episodes-per-batch and num-workers (keep similar to Phase 1 — 16 workers)
- Whether to use the `dt_substep` from the env config or hardcode 0.001s for t_start computation
- Exact shape/indexing of the checkpoint array (flattened by episode×step or nested)
- Whether `OverlappingPairDataset` does lazy vs eager loading of batch files

</decisions>

## Existing Code Insights

### Reusable Assets
- `collect_data.py:_worker_fn()`: Worker loop is the right place to change from 500→2000 substeps; inner `env.step()` call needs to be replaced with a manual substep loop with checkpoint saves
- `collect_data.py:SobolActionSampler`: Keep as-is — still used for action sampling
- `collect_data.py:perturb_rod_state()`: Keep as-is — still called after reset; only `perturb_omega_std` changes
- `collect_data.py:_find_next_batch_idx()`: Keep as-is — batch indexing logic unchanged
- `state.py:encode_phase()` / `encode_phase_batch()`: These encode global phase; need new function `encode_per_element_phase()` alongside
- `state.py:action_to_omega()`: Used to compute ω from action; unchanged, reused in dataset class
- `state.py:INPUT_DIM = 131`: Must be updated to 189
- `dataset.py:SurrogateDataset`: Keep but mark deprecated; `OverlappingPairDataset` is the new primary class
- `dataset.py:compute_density_weights()`: Can be reused with the new dataset class

### Established Patterns
- Dataclass config with CLI overrides (`DataCollectionConfig`, `collect_data.py:parse_args()`)
- Atomic batch saves: tmp file + `os.replace()` — keep this pattern
- Per-worker Sobol seed (`config.seed + worker_id * 1000`) — keep
- OpenBLAS/OMP/MKL thread limiting via env vars at module top — keep

### Integration Points
- `locomotion_elastica/env.py`: The 2000-substep inner loop needs to access the PyElastica rod's internal integration; env.step() currently runs 500 substeps internally — we need to either call step() 4 times per collection step (each internally running 500 substeps with the SAME action), or unwrap the substep loop. Calling env.step(action) 4 times and saving intermediate states is cleaner.
  - Actually: calling `env.step(action)` 4 times IS the "2000 substep" approach if each step runs 500 substeps. States at steps 0, 1, 2, 3, 4 of the RL macro-step (each 500 Elastica substeps apart) — this gives 5 checkpoints, not 201.
  - TRUE 2000 substeps with K=10 granularity requires unwrapping the env's inner loop.
  - **Alternate interpretation**: The user may mean 4 consecutive RL steps (4×500=2000 substeps) with the same action. Then checkpoints = the RL-step boundaries = 5 states, 4 valid 500-substep pairs. Simpler but fewer pairs.
  - **MUST CLARIFY before planning**: Does "2000 substeps, K=10" mean unwrapping PyElastica's internal loop, or does it mean running env.step() 4 times with the same action and saving at each macro-step boundary?
- `aprx_model_elastica/state.py:INPUT_DIM`: Update from 131 to 189 — this propagates to `model.py:SurrogateModel` first layer input dim and `train_surrogate.py` everywhere INPUT_DIM is referenced

### Key Implementation Risk
- PyElastica does not expose intermediate substep states via `env.step()`. To truly save every 10 substeps, we need to call the internal `timestepper.do_step()` or equivalent 200 times manually. This requires importing PyElastica internals and is fragile.
- Simpler alternative: treat each RL macro-step boundary (every 500 substeps) as the checkpoint. With 4 consecutive env.step(same_action) calls, we get 5 states and 4 valid pairs. Fewer pairs but no PyElastica internal API dependence.
- This decision (unwrap internal loop vs 4×env.step) is the primary planning risk for Phase 2.1.

</code_context>

<specifics>
## Specific Ideas

- 25 GB target dataset (vs 2.3 GB Phase 1) — ~10x more data by volume
- Per-element = 20 elements (not 21 nodes) — curvature is an element quantity in PyElastica (kappa.shape = (n_elems-1,) = (19,) for rest_kappa, but angular velocity and yaw are defined on 20 elements)
- Archive Phase 1 data to `data/surrogate_v1/` before writing new data to `data/surrogate/`
- Keep perturb_omega_std as a config field so it can be tuned without code change

</specifics>

<deferred>
## Deferred Ideas

- Finer dt (reduce substeps_per_action from 500 to 100): mentioned in omega_z issue as option 5. Not doing in Phase 2.1 — the overlapping pair approach addresses the data diversity issue differently. Revisit if Phase 4 trajectory validation still shows poor omega_z after 2.1+3.1.
- K-step sequence windows for history-conditioned training: save consecutive transitions for the HistorySurrogateModel in Phase 3.1. The raw checkpoint format partially supports this but the history model is out of scope here.

</deferred>
