## Summary

Phase 02.3 requires collecting snake robot dynamics data from the DisMech physics backend for surrogate model training. A direct code-level comparison of the existing Elastica pipeline (`papers/aprx_model_elastica/`) and the already-scaffolded DisMech pipeline (`papers/aprx_model_dismech/`) reveals that **the DisMech pipeline is already 80-90% implemented**. The scaffolding from quick task 260316-s3f created `collect_data.py`, `collect_config.py`, `state.py`, `dataset.py`, `model.py`, `env.py`, `health.py`, `monitor.py`, `preprocess_relative.py`, `train_config.py`, `train_surrogate.py`, and `validate.py` -- all adapted for DisMech.

The key finding is that **the state vector format is intentionally identical** between Elastica and DisMech: both produce 124-dim flat vectors with the same layout (pos_x(21), pos_y(21), vel_x(21), vel_y(21), yaw(20), omega_z(20)). This was a deliberate design decision -- `aprx_model_dismech/state.py` extracts the 2D (x, y) projection from DisMech's 3D state vectors using finite differences of segment tangents to derive yaw and angular velocity, producing the exact same representation.

**Primary recommendation:** The existing `papers/aprx_model_dismech/collect_data.py` needs targeted fixes (multi-step collection, force collection, worker respawn, W&B integration) and a smoke test before launching a full-scale collection run. The same `FlatStepDataset` format can be reused. The surrogate model architecture is identical (same 124/128/193-dim pipeline).

## Elastica vs DisMech Comparison

### Time Integration (CRITICAL DIFFERENCE)

| Property | Elastica | DisMech |
|----------|----------|---------|
| **Integration method** | Explicit symplectic (PositionVerlet / PEFRL) | Implicit Euler (Newton solver) |
| **Integration timestep** | dt_substep = 0.001s | dt = 0.05s (single step) |
| **Substeps per RL step** | 500 (500 x 0.001s = 0.5s) | 1 (1 x 0.05s = 0.05s) |
| **RL step duration** | 0.5s per action | 0.05s per action |
| **DT_CTRL constant** | 0.5s (state.py line 304) | 0.05s (state.py line 336) |
| **Numerical stability** | Conditional (CFL-limited) | Unconditionally stable |
| **Cost per RL step** | 500 Python-loop substeps | 1 Newton solve (C/Python) |

**Impact on data collection:** DisMech takes one implicit step per RL action (dt=0.05s). Elastica takes 500 explicit substeps per RL action (dt_substep=0.001s, total 0.5s). This means:
1. DisMech RL steps cover 10x less physical time (0.05s vs 0.5s)
2. State deltas will be ~10x smaller in magnitude per step
3. The n_cycles encoding is different (frequency * 0.05 vs frequency * 0.5)
4. Collection speed per transition should be faster (1 Newton solve vs 500 substep iterations)

### State Vector (IDENTICAL FORMAT)

| Property | Elastica | DisMech |
|----------|----------|---------|
| **Dimensionality** | 124 | 124 |
| **Layout** | pos_x(21), pos_y(21), vel_x(21), vel_y(21), yaw(20), omega_z(20) | SAME |
| **Node count** | 21 (20 segments + 1) | 21 (20 segments + 1) |
| **Element count** | 20 | 20 |
| **Joint count** | 19 | 19 |
| **Coordinate system** | 2D projection from 3D Cosserat rod | 2D projection from 3D DER |
| **Position source** | `rod.position_collection[0:2, :]` | `q[:63].reshape(21,3)[:, 0:2]` |
| **Velocity source** | `rod.velocity_collection[0:2, :]` | `u[:63].reshape(21,3)[:, 0:2]` |
| **Yaw source** | `arctan2(rod.tangents[1,:], rod.tangents[0,:])` | `arctan2(dy_seg, dx_seg)` finite diff |
| **omega_z source** | `rod.omega_collection[2, :]` | Cross-product: `(dx*dv_y - dy*dv_x) / seg_len_sq` |
| **Relative state** | 128-dim (same transform) | 128-dim (same transform) |
| **Model input** | 193-dim (128 + 5 + 60) | 193-dim (128 + 5 + 60) |

**Key difference in omega_z:** Elastica provides omega_z directly from the rod's angular velocity array (`rod.omega_collection[2, :]`). DisMech computes it via finite differences of the velocity field using the cross-product formula. Both produce the same physical quantity (element angular velocity about z-axis), but the DisMech computation may have slightly different numerical characteristics due to the discrete approximation.

| Property | Elastica | DisMech |
|----------|----------|---------|
| **Dimensionality** | 5 | 5 |
| **Layout** | [amplitude, frequency, wave_number, phase_offset, turn_bias] | SAME |
| **Range** | [-1, 1] normalized | [-1, 1] normalized |
| **Frequency range** | (0.5, 3.0) Hz | (0.5, 3.0) Hz |

### Force/Torque Data (MAJOR DIFFERENCE)

| Property | Elastica | DisMech |
|----------|----------|---------|
| **Available?** | Yes -- rod exposes force arrays | No -- returns zeros (placeholder) |
| **external_forces** | `rod.external_forces` (3, 21) | `np.zeros((3, 21))` |
| **internal_forces** | `rod.internal_forces` (3, 21) | `np.zeros((3, 21))` |
| **external_torques** | `rod.external_torques` (3, 20) | `np.zeros((3, 20))` |
| **internal_torques** | `rod.internal_torques` (3, 20) | `np.zeros((3, 20))` |
| **Force in config** | `collect_forces: bool = False` but available | `collect_forces: bool = False` and N/A |

**Impact:** Force data was collected in Phase 02.2 (Elastica) for future analysis but is NOT used as surrogate model input. Since DisMech does not expose per-node force arrays, the DisMech collection should set `collect_forces=False` (already the default). Forces are irrelevant for surrogate training.

### Curvature Control (DIFFERENT MECHANISM, SAME INTERFACE)

| Property | Elastica | DisMech |
|----------|----------|---------|
| **Control target** | `rod.rest_kappa[0, :]` (Cosserat rest curvature) | `bend_springs.nat_strain[:, 0]` (DER natural strain) |
| **Shape** | (3, n_elements-1) = (3, 19) | (n_springs, 2) ~ (19, 2) |
| **Application** | Direct write to rod | Direct write to bend springs |
| **CPG → curvature** | Via `_apply_serpenoid()` in env | Via `_generate_serpenoid_curvatures()` in collect_data |
| **Effect** | Elastic restoring force toward rest curvature | Elastic energy minimized at natural strain |

- State magnitudes may differ (different integration, different physics model)
- Delta magnitudes will differ significantly (0.05s steps vs 0.5s steps means ~10x smaller deltas)
- A model trained on Elastica data CANNOT be used on DisMech data without retraining

### Anti-Patterns to Avoid
- **Reusing Elastica normalizer for DisMech data:** The state magnitudes and delta distributions are different between backends. Always compute fresh normalization statistics from DisMech data.
- **Assuming same RL step duration:** Elastica DT_CTRL=0.5s, DisMech DT_CTRL=0.05s. The per-element phase encoding uses DT_CTRL, which is correctly set in each package's `state.py`.
- **Collecting force data from DisMech:** The current `pack_forces()` returns zeros. Do not waste disk space or I/O time on placeholder force data.
- **Using Elastica environment wrapper for DisMech collection:** The DisMech collect_data.py correctly uses `SnakeRobot` directly (not a TorchRL env wrapper). This is intentional -- data collection bypasses the env's observation/reward logic.

## Common Pitfalls

### Pitfall 1: DisMech Solver Convergence Failures
**What goes wrong:** DisMech's implicit Euler solver may fail to converge for extreme curvature inputs or perturbed states, raising ValueError.
**Why it happens:** The Newton solver has max_iter=25 and tol=1e-4. Extreme curvatures (near clip limit of +/-10 rad/m) combined with large perturbations can cause the Jacobian to be ill-conditioned.
**How to avoid:** The `SnakeRobot.step()` already catches ValueError and prints a warning. The collection loop should treat convergence failure as a NaN episode (discard and continue). Ensure `validate_episode_finite()` is called after every step.
**Warning signs:** High NaN discard rate (>10%), worker stalls.

### Pitfall 2: Wrong DT_CTRL Value
**What goes wrong:** Using Elastica's DT_CTRL=0.5s for DisMech data causes phase encoding to be 10x too fast.
**Why it happens:** Copy-paste from Elastica code without updating the time constant.
**How to avoid:** The DisMech state.py already sets `DT_CTRL = 0.05` (correct). Verify this value matches `config.physics.dt = 0.05` in DismechConfig.
**Warning signs:** n_cycles encoding values look wrong (too many cycles per step).

### Pitfall 3: serpenoid_time Not Advancing (Current Bug)
**What goes wrong:** The existing DisMech collect_data.py creates a fresh SnakeRobot for each transition and sets `serpenoid_time = 0.0`. This means t_start is always 0.0 in the saved data.
**Why it happens:** Unlike Elastica's multi-step episodes (which run 100+ steps per episode), the current DisMech code collects one step per fresh robot. The serpenoid_time never advances.
**How to avoid:** Either (a) collect multi-step episodes where serpenoid_time advances naturally, or (b) randomize the initial serpenoid_time for each single-step collection. Option (a) is preferred for consistency with Elastica pipeline.
**Warning signs:** All t_start values in batch files are 0.0.

### Pitfall 4: omega_z Numerical Noise
**What goes wrong:** DisMech's omega_z is computed via finite differences of the velocity field, which can amplify numerical noise for nearly-straight rods where segment length is small relative to numerical precision.
**Why it happens:** The formula `(dx * dv_y - dy * dv_x) / (dx^2 + dy^2 + eps)` has eps=1e-10, but for a straight rod with segment length ~0.025m, the denominator is already ~6e-4, providing decent numerical conditioning.
**How to avoid:** The eps=1e-10 in the existing code handles degenerate cases. Monitor omega_z statistics during collection -- if std is much larger than Elastica's, it indicates a numerical issue.
**Warning signs:** omega_z values orders of magnitude larger than expected (~1-10 rad/s operational range).

### Pitfall 5: MKL Thread Conflict
**What goes wrong:** DisMech (and dismech-rods) use PARDISO solver which requires single-threaded MKL. If torch or numpy initializes MKL in multi-threaded mode first, symbolic factorization errors occur.
**Why it happens:** MKL threading state is process-global and set at first use.
**How to avoid:** The existing collect_data.py sets `os.environ["MKL_NUM_THREADS"] = "1"` at the top of the file, before any imports. The DisMech snake_robot.py also sets this. Multiprocessing workers inherit the env vars.
**Warning signs:** Segfaults or "PARDISO: symbolic factorization error" messages.

### Pitfall 6: Dataset Size Mismatch Expectations
**What goes wrong:** DisMech steps are 10x faster in wall time (no substep loop) but produce 10x less physical time per transition. If you target the same number of transitions as Elastica (5M), you get 10x less simulated time coverage.
**Why it happens:** Each DisMech transition covers 0.05s vs Elastica's 0.5s.
**How to avoid:** Either (a) collect 10x more transitions to match total simulated time, or (b) increase steps_per_run to collect multi-step episodes, or (c) accept the shorter time window per transition and let the model learn finer-grained dynamics. Option (c) is likely sufficient since the surrogate only needs to predict one RL step ahead.
**Warning signs:** Surrogate model trained on DisMech data has poor long-horizon rollout accuracy.

## Code Examples

### Example 1: RodState2D.pack_from_dismech() (existing, verified)
```python
# From papers/aprx_model_dismech/state.py
# Extracts 124-dim 2D state from DisMech's 3D state vectors
q = snake_robot._dismech_robot.state.q
positions = q[:3 * NUM_NODES].reshape(NUM_NODES, 3)
pos_x = positions[:, 0].copy()  # (21,)
pos_y = positions[:, 1].copy()  # (21,)

u = snake_robot._dismech_robot.state.u
velocities = u[:3 * NUM_NODES].reshape(NUM_NODES, 3)
vel_x = velocities[:, 0].copy()  # (21,)
vel_y = velocities[:, 1].copy()  # (21,)

# Yaw from finite differences of segment tangent vectors
dx = positions[1:, 0] - positions[:-1, 0]  # (20,)
dy = positions[1:, 1] - positions[:-1, 1]  # (20,)
yaw = np.arctan2(dy, dx).copy()  # (20,)

# omega_z from cross product of tangent and velocity difference
dv_x = velocities[1:, 0] - velocities[:-1, 0]
dv_y = velocities[1:, 1] - velocities[:-1, 1]
seg_len_sq = dx**2 + dy**2 + 1e-10
omega_z = (dx * dv_y - dy * dv_x) / seg_len_sq  # (20,)
```

### Example 2: Curvature Generation for DisMech (existing, verified)
```python
# From papers/aprx_model_dismech/collect_data.py
def _generate_serpenoid_curvatures(action, t, num_joints=19, freq_range=(0.5, 3.0)):
    A = (float(np.clip(action[0], -1.0, 1.0)) + 1.0) / 2.0 * 5.0
    omega = 2.0 * np.pi * frequency
    k = 2.0 * np.pi * wave_number
    phi = float(np.clip(action[3], -1.0, 1.0)) * np.pi
    turn_bias = float(np.clip(action[4], -1.0, 1.0)) * 2.0

    s_j = np.linspace(0.0, 1.0, num_joints, dtype=np.float64)
    curvatures = A * np.sin(k * s_j + omega * t + phi) + turn_bias
    return curvatures.astype(np.float64)
```

   - Recommendation: Start with 1M transitions for a smoke/training test. Scale to 5M if surrogate quality is good. The smaller delta magnitudes may actually be easier to learn.

2. **Should DisMech steps_per_run be increased?**
   - What we know: Current config has steps_per_run=1. Elastica Phase 02.2 also used steps_per_run=1 but with 500 substeps per step (0.5s physical time). DisMech at steps_per_run=1 gives only 0.05s per step.
   - What's unclear: Whether increasing steps_per_run to 10 (to match 0.5s physical time per episode) would improve data diversity or if single-step collection is sufficient.
   - Recommendation: Keep steps_per_run=1 for initial collection. The surrogate model predicts one step ahead regardless of step duration. Multi-step data would only matter for trajectory-based training losses.

3. **Can Elastica and DisMech surrogates be cross-compared?**
   - What we know: Same 124-dim state format, same action space, same per-element phase encoding. Different physics models and time scales.
   - What's unclear: Whether the surrogate trained on DisMech data produces qualitatively similar dynamics to the Elastica surrogate.
   - Recommendation: This is a Phase 9 question. For Phase 02.3, focus on collecting good DisMech data. Cross-comparison is future work.
