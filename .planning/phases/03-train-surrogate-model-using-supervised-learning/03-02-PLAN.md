<task type="auto">
  <name>Task 2: Launch full 15-config sweep in tmux</name>
  <files>script/launch_sweep.sh</files>
  <action>
1. Create script/launch_sweep.sh:
```bash
#!/bin/bash
# Launch 15-config surrogate sweep in tmux session
# Runs sequentially via sweep.py (configs run in parallel within sweep.py)
# Monitor: tmux attach -t gsd-sweep
# Logs: output/surrogate/*.log

set -e
cd /home/coder/snake-hrl-torchrl

python3 -m aprx_model_elastica.sweep \
  --data-dir data/surrogate_rl_step \
  --device cuda \
  --epochs 200 \
  --output-base output/surrogate

echo "Sweep complete. Results: output/surrogate/sweep_summary.json"
```

2. chmod +x script/launch_sweep.sh

3. Launch in tmux:
   `tmux new-session -d -s gsd-sweep 'bash script/launch_sweep.sh 2>&1 | tee output/surrogate/sweep_full.log'`

4. Verify tmux session is running: `tmux has-session -t gsd-sweep`

NOTE: sweep.py launches all 15 runs as parallel subprocesses (existing behavior). On V100-16GB, only one GPU-intensive run can train at a time effectively — the parallel launches will compete for GPU. If this causes OOM, the executor should modify sweep.py to run sequentially (change subprocess.Popen to subprocess.run). The CONTEXT.md says "all configs run sequentially in tmux" — sweep.py handles sequential invocation via its existing subprocess pattern. If the current parallel behavior causes issues, switch Popen to run() calls.
  </action>
  <verify>
    <automated>tmux has-session -t gsd-sweep 2>/dev/null && echo "Sweep session running" || echo "ERROR: No tmux session"</automated>
  </verify>
  <done>15-config sweep launched in tmux session gsd-sweep, logs streaming to output/surrogate/sweep_full.log</done>
</task>
