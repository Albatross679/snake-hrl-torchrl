Modify `MMRKHSTrainer` to implement the 5 notebook mechanics. All changes are gated by config flags.

**In `__init__`:**
- Add `self._global_batch_idx = 0` counter (tracks total batches across training for adaptive schedules).

**In `train()`:**
- After `metrics = self._update(batch)`, increment `self._global_batch_idx += 1`.

**In `_update()`:**

1. **Adaptive eta** (gated by `self.config.eta_schedule`):
   At the top of `_update()`, before the epoch loop:
   ```python
   if self.config.eta_schedule:
       k = self._global_batch_idx
       eta_effective = self.config.eta * (k + 1) ** self.config.eta_exponent
   else:
       eta_effective = self.config.eta
   ```
   Then in the loss computation, replace `(1.0 / self.config.eta)` with `(1.0 / eta_effective)`.

2. **Adaptive beta** (gated by `self.config.beta_schedule`):
   Inside the mini-batch loop, after advantage normalization:
   ```python
   if self.config.beta_schedule:
       k = self._global_batch_idx
       beta_effective = advantage.abs().max().item() / max(math.sqrt(k + 1), 1.0)
   else:
       beta_effective = self.config.beta
   ```
   Use `beta_effective` instead of `self.config.beta` in the loss.

3. **Tighter exponent clipping** (always active, just uses config value):
   Change `log_ratio = (log_prob_new - log_prob_old).clamp(-20.0, 20.0)` to:
   ```python
   log_ratio = (log_prob_new - log_prob_old).clamp(-self.config.exponent_clip, self.config.exponent_clip)
   ```
   Default 20.0 preserves current behavior; user sets 2.0 for notebook-like tightness.

4. **Inner MM iterations** (gated by `self.config.inner_mm_iterations > 1`):
   Wrap the forward-loss-backward block in the mini-batch loop with:
   ```python
   for mm_iter in range(self.config.inner_mm_iterations):
   ```
   On each inner iteration:
   - Re-forward the actor to get fresh `log_prob_new`, `new_loc`, `new_scale`
   - Recompute ratio, MMD, KL, critic loss, total loss
   - Backward + optimizer step
   - The key: on iterations after the first, `log_prob_old` stays fixed (from collection) but the policy has already been updated by prior inner steps, so the ratio naturally evolves

   Track `actual_updates` correctly (increment per inner iteration).

5. **Direct kernel correction** (gated by `self.config.kernel_correction`):
   After computing `mmd`, add a kernel correction term:
   ```python
   if self.config.kernel_correction:
       # Analogous to notebook's R @ (pi_old - pi) term
       # In mean-parameter space: correction = ||old_loc - new_loc||^2
       # Weighted by advantages (like R_a_T_pi in notebook)
       loc_diff = (old_loc - new_loc)  # [batch, action_dim]
       kernel_corr = (loc_diff.pow(2).sum(dim=-1) * advantage.abs()).mean()
       kernel_corr = self.config.kernel_correction_weight * kernel_corr
   else:
       kernel_corr = torch.tensor(0.0, device=batch.device)
   ```
   Add `kernel_corr` to the total loss and log it as `metrics["kernel_correction"]`.

**Update the total loss line to:**
```python
total_loss = (
    -surr_advantage.mean()
    + beta_effective * mmd
    + (1.0 / eta_effective) * kl
    + self.config.value_coef * critic_loss
    + kernel_corr
)
```

**Add new metrics to accumulation:**
- `"kernel_correction"` in the metrics dict and accumulated_keys set
- `"eta_effective"` and `"beta_effective"` as non-accumulated (just last value)

**In `_log_metrics()`:**
- Add `"train/eta_effective"` and `"train/beta_effective"` and `"train/kernel_correction"` to the wandb_log dict if present in metrics.

Do NOT touch `_compute_mmd_penalty` — it stays as-is. The kernel correction is a separate additive term.
