# Project Instructions

## Project Reference

For project details, architecture, and background information, refer to @documents/overview.tex. This file contains comprehensive documentation about the project.

## Documentation Requirements

IMPORTANT: You MUST document all work in the `documents/` folder. Based on the instructions from the user, update these files after completing any task:

### 1. documents/logs.md
Log every task or change session. Use this format:
```markdown
## log N

**Date:** YYYY-MM-DD

**Task:** [Brief description of what was requested]

### changes made
- [File changed]: [What was modified]
- [File changed]: [What was modified]

### summary
[Brief summary of what was accomplished]
```

### 2. documents/experiments.md
Document any experiments, tests, or investigations. Use this format:
```markdown
## Experiment N: [Title]

**Date:** YYYY-MM-DD

**Objective:** [What you're trying to learn/test]

### Setup
[Configuration, parameters, environment]

### Results
[Observations, data, outputs]

### Conclusions
[What was learned, next steps]
```

### 3. documents/issues.md
Track bugs, errors, or problems encountered. Use this format:
```markdown
## Issue #N: [Title]

**Date:** YYYY-MM-DD

**Status:** Open | In Progress | Resolved

**Component:** [File or system affected]

### Description
[What the issue is]

### Steps to Reproduce
[How to trigger the issue]

### Workaround / Resolution
[How to fix or avoid it]
```

## When to Document

- **logs.md**: After ANY code changes, file modifications, or completed tasks
- **experiments.md**: When running simulations, testing configurations, or investigating behavior
- **issues.md**: When encountering errors, bugs, or unexpected behavior

## Documentation Workflow

1. Before starting work, check the current log number in `documents/logs.md`
2. After completing the task, add a new log entry
3. If experiments were run, add to `documents/experiments.md`
4. If issues were found, add to `documents/issues.md`
